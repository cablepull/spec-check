spec-check

Spec-driven development enforcement for LLMs and humans alike — runs entirely offline via MCP.

MCP server TypeScript DuckDB · Parquet 26 tools offline · no telemetry
Get started GitHub

The problem

LLMs are powerful but structurally inconsistent. Given the same task twice, a model may produce a thorough specification one time and skip it entirely the next. Without enforcement, spec-driven methodology becomes aspirational rather than operational.

The discipline degrades under time pressure. Codebases accumulate undocumented decisions, untracked bugs, and architecture nobody can explain.

spec-check acts as a local gatekeeper. An LLM calls it, receives a deterministic verdict with specific violations, and must satisfy those violations before advancing. The verdict is structured JSON — the LLM can self-correct autonomously.

Every result is persisted locally as Parquet. Compliance trends become visible and improvable over time — per project and across all projects.

How it works

Projects are expected to maintain five spec artifacts. spec-check verifies them through a sequence of gates. Each gate returns structured pass/fail criteria with evidence and fix instructions.

# Agent workflow (typical)

begin_session      →  returns phase, obligations, and must_call_next
run_all            →  G1 → G2 → G3 → G4 → G5, stops at first BLOCK
                       fix violations, then:
gate_check:G2      →  targeted recheck after fixing a single gate
run_all            →  full sweep once individual gate passes
diff_check         →  analyse git diff, identify re-run requirements
complexity         →  CC, cognitive, nesting, length analysis
check_mutation_score →  mutation testing with trend detection
metrics            →  query per-project compliance history
close_session      →  persist final state
After any passing individual gate, must_call_next always resolves to run_all — not a chain of individual checks. Targeted gate_check calls are for fixing a known single violation, not for initial sweeps.

Response structure

Every tool returns the same envelope:

{
  "status": "PASS" | "BLOCKED" | "FAILING" | "PASSING_WITH_WARNINGS",
  "criteria": [
    {
      "id": "R-6",
      "status": "VIOLATION",
      "detail": "requirements.md has no numbered rules in any Feature section.",
      "evidence": ["F-1: Missing Rule R-n pattern"],
      "fix": "Add ### Rule R-N: Validate ... lines under each Feature heading."
    }
  ],
  "workflow": {
    "must_call_next": ["run_all"],
    "should_call_metrics": false,
    "blocked": false
  }
}

All 26 tools

Each tool is callable from any MCP client. All accept a path argument pointing to the project root.

run_all

Run all five gates sequentially. Stops at the first BLOCKED gate and returns next-step instructions.

gate_check

Run a single gate (G1–G5). For targeted re-checks after fixing a specific violation.

diff_check

Analyse git diff to identify which gates need re-running. Reports ADR trigger status on architectural changes.

validate_artifact

Validate a single file (intent, requirements, design, tasks, story, adr, rca) with full criteria output.

complexity

Cyclomatic complexity, cognitive complexity, nesting depth, and function length. Uses bundled AST parsers and lizard.

check_mutation_score

Mutation testing via Stryker, mutmut, go-mutesting, or cargo-mutants. Returns score, survivors, and trend detection.

metrics

Query stored compliance metrics: gate pass rates, compliance score, CC history, mutation trend per project.

get_rollup

Cross-project compliance rankings, model comparisons, and agent performance statistics.

scaffold_spec

Generate spec file templates with guidance patterns and optionally write them to disk.

get_spec_guide

Reference guide teaching how to write passing spec files on the first attempt.

get_protocol

Return the full self-describing protocol: 5-gate methodology, tool signatures, severity levels.

check_reconciliation

Reconcile README claims against repository artifacts (RC-1) and verify task file references (RC-2).

check_evidence

Verify release artifacts have verification evidence (EV-1) and performance components have benchmarks (EV-2).

check_assumptions

Validate assumptions table: presence, shape, hedging language, and orphan invalidation detection.

track_assumption

Register a new assumption in the registry. Assumptions must be tracked before they can be invalidated.

invalidate_assumption

Mark an assumption as invalidated, triggering supersession of affected artifacts.

list_assumptions

List all tracked assumptions for a project path with status and history.

get_assumption_metrics

Assumption invalidation rates, category breakdown, and model-level accuracy statistics.

get_supersession_history

Supersession events for a project, optionally filtered to a date range.

check_dependencies

Report installed/missing analysis tools, what each enables, and available install commands.

install_dependency

Install a named dependency using the highest-priority available package manager.

begin_session

Register an agent session and receive initial workflow phase, obligations, and must_call_next.

report_agent_state

Persist caller-reported workflow state so the server can compute next required actions.

get_next_action

Return computed next checks, blocking prerequisites, and metrics obligations for the current agent.

list_agent_state

List latest workflow state for all agents in a project or session.

close_session

Mark the current session complete and persist final state.

Metrics and compliance history

Every tool call writes a Parquet record to ~/.spec-check/data. DuckDB queries the records on demand — no external database, no always-on process.

Gate heatmap

The compliance dashboard renders a gate heatmap: rows are G1–G5, columns are run iterations. Each column represents one run_all sweep (not individual gate records), so you can see how compliance evolved across iterations.

Each column = one run_all call. Multiple iterations on the same day show HH:MM labels.

Configuration

Place spec-check.config.json in the project root. All fields are optional — defaults apply if absent.

{
  "thresholds": {
    "CC-1": 10,      // max CC per function
    "CC-4": 4,       // max nesting depth
    "MT-1": 80,      // project mutation score floor (%)
    "MT-2": 90       // spec-critical function floor (%)
  },
  "compliance_weights": {
    "G1": 0.15, "G2": 0.30, "G3": 0.20,
    "G4": 0.15, "G5": 0.20
  },
  "metrics": {
    "db_path": "~/.spec-check/data",
    "retention_days": 365
  },
  "mutation": {
    "enabled": true,
    "incremental": true,
    "triggers": { "default": "pre_merge" }
  }
}

Get started

1

Install

$ npm install -g spec-check
# or use without installing:
$ npx spec-check
2

Add to Claude Desktop

Edit ~/Library/Application Support/Claude/claude_desktop_config.json:

{
  "mcpServers": {
    "spec-check": {
      "command": "npx",
      "args": ["-y", "spec-check"],
      "env": {
        "LLM_PROVIDER": "anthropic",
        "LLM_MODEL": "claude-sonnet-4-6"
      }
    }
  }
}

Restart Claude Desktop after saving. The server connects via stdio — no port, no daemon.

3

Scaffold your spec files

Ask Claude to scaffold your spec. It will call scaffold_spec with your project path, generating templates for all five artifacts.

// Example prompt
"scaffold spec files for my project at /path/to/project"
4

Run your first gate check

// Ask Claude:
"run all gates on /path/to/project"

// The server calls run_all and returns:
{
  "status": "BLOCKED",
  "gates": [
    {
      "gate": "G1",
      "status": "BLOCKED",
      "criteria": [{
        "id": "I-1",
        "status": "BLOCK",
        "detail": "intent.md not found",
        "fix": "Create intent.md with Problem and Intent sections"
      }]
    }
  ]
}
5

Optional: install analysis tools

// Check what's available:
"check what analysis dependencies are installed"

// Install lizard for cognitive complexity:
$ pip install lizard

// Install Stryker for mutation testing (TypeScript/JS):
$ npm install --save-dev @stryker-mutator/core
